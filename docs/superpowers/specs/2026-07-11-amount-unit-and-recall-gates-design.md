# 成交额单位修复与召回门槛调整设计

## 1. 背景

Stock Lobster 当前直接消费外部事实生产者的 `token_daily_details.amount`，并据此计算：

- `avg_amount_20d`
- `amount_ratio_20d`
- 最低流动性门槛
- 突破放量确认

只读排查确认，`token_daily_details.amount` 在 2026-04-27 发生全市场单位切换：

- 2026-04-24 及以前，`amount / (close * vol)` 约为 `0.1`，符合 Tushare 日线接口的“千元”口径。
- 2026-04-27 及以后，该比值约为 `1000`，相对前一交易日放大约 `10000` 倍。
- 外部生产任务当前执行 `float(row["amount"]) * 10000`，与源接口单位不一致。

这会同时污染绝对成交额门槛和单位切换后约 20 个交易日的滚动量比。策略参数不能在该数据状态下继续校准。

## 2. 已批准决策

采用方案 A：

```text
token_daily_details.amount = Tushare daily.amount 原始值，单位为千元
```

理由：

- `token_daily_details` 是 `source_raw` 表，应保留上游数据源的原始字段语义。
- 避免使用“实际元乘 10”这类非标准且难以审计的隐式单位。
- 历史数据在 2026-04-24 及以前已经符合该口径，修复范围较小。
- 绝对成交额阈值可以通过显式单位换算表达，不需要改写源事实含义。

## 3. 系统边界

### 3.1 外部事实生产者负责

外部事实生产者负责：

- 修正新增日线数据的 `amount` 入库逻辑。
- 修复 2026-04-27 起已写入的错误历史事实。
- 重算所有依赖成交额的事实统计和基础指标。
- 发布可机读的字段单位和质量状态。
- 增加单位漂移检查，阻断异常数据发布。

### 3.2 Stock Lobster 负责

Stock Lobster 只负责：

- 在 L0 数据契约中声明并校验 `amount_unit=thousand_cny`。
- 在 L1/L2 中使用统一单位计算确定性指标。
- 将人民币金额阈值显式换算为千元后比较。
- 数据质量未通过时停止生成相关标签和信号。
- 在事实数据修复后重新运行样本分析、扫描和回测。

Stock Lobster 不得在查询或策略代码中按日期偷偷除以或乘以常数修补事实数据。

## 4. 外部数据修复设计

### 4.1 新增数据修复

将日线采集任务从：

```python
"amount": float(row["amount"]) * 10000
```

调整为：

```python
"amount": float(row["amount"])
```

同时在表注册表、数据产品注册表和字段文档中声明：

```text
amount_unit: thousand_cny
volume_unit: lot
```

### 4.2 历史事实修复

历史修复边界为：

```text
trade_date >= 20260427
```

修复动作：

```text
amount = amount / 10000
```

执行前必须保存受影响行数、日期范围和修复前校验摘要。修复应在事务或可恢复备份保护下执行，完成后按交易日检查：

- 行数不变。
- 主键不变。
- OHLC、`vol` 等非目标字段不变。
- `amount` 非空率和正值率不下降。
- `amount / (close * vol)` 的市场中位区间恢复到千元/手口径。

### 4.3 衍生表重算

至少重算：

- `amount_daily_statistic`
- `amount_weekly_statistic`
- 所有含 `amount`、`avg_amount`、`amount_ratio` 的月度统计
- `pub_stock_daily_indicator` 中由上述表发布的成交额指标
- 受成交额质量状态约束的发布批次

不能只修源表而保留旧的滚动统计结果。

## 5. 数据质量门

外部生产者增加两类检查。

### 5.1 字段级单位检查

对 `amount > 0`、`vol > 0`、`close > 0` 的样本计算：

```text
amount_to_notional_ratio = amount / (close * vol)
```

在 `amount=千元`、`vol=手` 的口径下，该比值应大致围绕 `0.1`。质量门使用宽容区间和市场分位数，避免因涨跌和成交均价差异误报；若相邻交易日市场中位数发生数量级跳变，则发布失败。

### 5.2 滚动连续性检查

检查每日市场中位 `amount`、个股 `amount_ratio_20d` 高分位和跨日变化倍数。出现全市场同步放大百倍以上时，应标记数据批次失败，而不是继续生产衍生指标。

## 6. Stock Lobster 金额契约调整

当前“20 日平均成交额至少 2 亿元”在千元口径下应表示为：

```text
200,000 千元 = 200,000,000 元
```

因此现有 `min_avg_amount_20d=2_000_000_000` 需要改为 `200_000`，并删除“原始值约等于实际元乘 10”的说明。

金额参数应同时记录：

- 业务含义：人民币元。
- 输入字段单位：千元。
- 确定性换算公式。

L0 在单位元数据缺失或不匹配时应拒绝构建可用于信号生产的快照。

## 7. 量比口径调整

当前口径保持为兼容指标：

```text
amount_ratio_20d_inclusive = 当日成交额 / 包含当日在内的 20 日平均成交额
```

同时新增研究候选指标：

```text
amount_ratio_prev_20d = 当日成交额 / 前 20 个交易日平均成交额
```

二者必须使用相同的成交额单位。单位统一后量比本身与单位无关，但跨单位切换窗口必须整体作废并重算。

`1.5` 不再作为所有突破形态的统一召回硬门槛：

- 保留为 `volume_expansion_confirmed` 强放量标签。
- 在长期盘整突破中作为确认或加分项。
- 在回调再加速、MA10/MA20 稳步上行和趋势跟随子池中改为评分项。
- 低于 `1.0` 可以降权，但不得脱离形态上下文直接判负。

正式阈值必须在修复后的正负样本和历史事件回测上重新审批。

## 8. MA30 承接窗口调整

`ma30_hold_ratio_90d >= 0.75` 不再作为所有 pre-breakout 候选的统一硬门槛。

按召回子池使用：

| 子池 | MA30 口径 |
| --- | --- |
| 长期盘整突破 | 关注近期站回和平台结构，90 日承接只做诊断 |
| 回调再加速 | `30 日承接 >= 75%` 且 `60 日承接 >= 55%` |
| MA10/MA20 稳步上行 | 使用 60/90 日趋势质量评分，不要求单一硬阈值 |
| 趋势跟随 | 可保留 90 日承接作为成熟趋势质量项 |
| 早期反转 | 不要求此前 90 日站上 MA30，改用修复和反转确认 |

放宽近期承接会同时召回部分等待和负样本，因此必须配合负向结构标签：

- 大阳线后连续阴线回调。
- V 形修复过陡且缺少平台。
- 高位盘整时间不足。
- 均线黏合后破位或噪声过高。
- 加速后缺少休整。

## 9. 执行顺序

严格按以下顺序执行：

1. 修改外部生产者新增数据入库单位。
2. 增加单位质量检查和失败阻断。
3. 修复 `token_daily_details` 历史错误区间。
4. 重算成交额衍生表和发布产品。
5. 用 SQL 验证源表、衍生表、重复、缺失和单位连续性。
6. 更新 Stock Lobster 的 L0 契约和金额阈值。
7. 重新导出样本日期输入并运行现有策略。
8. 重新生成逐样本阻断报告。
9. 以 research-only 候选实现分子池量比和 MA30 规则。
10. 完成正负样本对比和历史事件回测后，再决定策略晋升。

## 10. 验收标准

事实数据修复验收：

- 2026-04-24 与 2026-04-27 前后单位连续。
- `amount / (close * vol)` 不再出现全市场约 10000 倍跳变。
- 源表修复范围和受影响行数可审计。
- 所有成交额衍生表已重算且日期覆盖完整。
- 重算后不存在主键重复、交易日缺失或空值异常。

策略侧验收：

- `avg_amount_20d` 的业务金额与千元字段换算一致。
- 量比跨单位切换窗口全部重算。
- 逐样本报告能够区分数据质量阻断、基础流动性阻断、形态阻断和触发阻断。
- 分子池候选保持 `research_only`，不直接替换当前生产策略。
- 正样本召回改善必须同时报告硬负、等待和低价值样本的误召回变化。
